1) КРАТКОЕ РЕЗЮМЕ

Quick-Job сейчас реализован как рабочий MVP платформы гарантированной подработки с эскроу-логикой.
Ключевая ценность уже работает:
- исполнитель не начинает без подтвержденных средств,
- заказчик не оплачивает без результата,
- платформа автоматически удерживает комиссию 5%.

Важно: на этапе MVP денежный ввод/вывод сделан в ручной операционной модели (через администратора), а не через Stripe/PayPal. Это осознанное бизнес-решение для быстрого запуска, демонстрации клиентам и первых сделок без долгой платежной интеграции.

2) ТЕХНИЧЕСКАЯ ОСНОВА (ЧТО УЖЕ СДЕЛАНО)

Backend:
- NestJS + Prisma + PostgreSQL (Supabase)
- JWT авторизация
- роли пользователей: ADMIN, CLIENT, WORKER
- API для заказов, откликов, эскроу, кошельков, пользователей

Frontend:
- Next.js (App Router) + Tailwind
- защищенные маршруты для авторизованных пользователей
- постоянная навигация (Navbar)
- страницы: dashboard, jobs, job details, profile, admin
- RU/EN локализация интерфейса

3) РОЛИ И ПРАВА

ADMIN:
- видит пользователей и их балансы,
- вручную зачисляет средства на баланс клиента после внешнего перевода,
- имеет обзор транзакций (глобальный ledger в текущем MVP),
- сопровождает ручные выплаты исполнителям вне платформы при необходимости.

CLIENT (заказчик):
- создает заказ и этапы (milestones),
- принимает/отклоняет отклики исполнителей,
- замораживает средства по этапам (escrow lock),
- подтверждает выполнение и инициирует выплату.

WORKER (исполнитель):
- отправляет отклики на заказы,
- после назначения выполняет работу,
- отправляет этап на проверку,
- получает выплату после подтверждения заказчиком.

4) МОДЕЛЬ ДЕНЕГ В MVP (КАК ЭТО РАБОТАЕТ В РЕАЛЬНОСТИ)

Вне платформы (реальные деньги):
1. Заказчик делает реальный перевод владельцу платформы/администратору.
2. Администратор вручную пополняет баланс заказчика в админ-панели.
3. После завершения работы (или набора работ) администратор вручную отправляет реальные деньги исполнителю.

Внутри платформы (цифровой ledger):
1. Деньги отображаются как доступный баланс заказчика.
2. При финансировании этапа сумма переносится в замороженный баланс.
3. При подтверждении выполнения сумма списывается из frozen:
   - 95% -> доступный баланс исполнителя,
   - 5% -> баланс платформы (комиссия).
4. Все движения фиксируются транзакциями.

Итог: ручной фиатный контур + автоматизированный внутренний учет и контроль сделки.

5) ПОЛНЫЙ ЖИЗНЕННЫЙ ЦИКЛ СДЕЛКИ

Шаг 1. Пополнение:
- ADMIN -> /admin -> Deposit Funds клиенту.
- У клиента увеличивается available balance.

Шаг 2. Создание заказа:
- CLIENT публикует job с бюджетом и этапами.

Шаг 3. Отклики:
- WORKER отправляет proposal (может предложить свою цену).

Шаг 4. Выбор исполнителя:
- CLIENT принимает нужный proposal.
- При принятии отклика согласованная сумма применяется к оплате (обновляется база для финансирования/выплаты).

Шаг 5. Финансирование этапа:
- CLIENT жмет Fund Milestone.
- available уменьшается, frozen увеличивается на сумму этапа.

Шаг 6. Сдача:
- WORKER жмет Submit Work.
- этап уходит в REVIEW.

Шаг 7. Подтверждение и выплата:
- CLIENT жмет Confirm & Pay.
- система делает release:
  - worker receives 95%,
  - platform fee 5%.
- этап становится COMPLETED, при завершении всех этапов job = COMPLETED.

6) ЭСКРОУ-ЛОГИКА И БЕЗОПАСНОСТЬ СДЕЛКИ

Что гарантируется текущей реализацией:
- без финансирования этапа исполнитель не может корректно пройти цикл до оплаты,
- деньги этапа блокируются в frozen, а не "исчезают",
- выплата идет только после подтверждения заказчика,
- комиссия платформы фиксируется автоматически в ledger.

Поддержка споров:
- milestone может быть переведен в DISPUTED,
- админ-разрешение: refund клиенту или release исполнителю (по бизнес-решению).

7) ТРАНЗАКЦИИ И ПРОЗРАЧНОСТЬ

Типы транзакций в системе:
- DEPOSIT
- ESCROW_LOCK
- RELEASE
- PLATFORM_FEE
- REFUND

Для администратора в текущей версии доступен глобальный обзор движения операций, что достаточно для ручного MVP-операционного контроля и демонстрации экономической модели платформы.

10) ТЕКУЩИЕ ОГРАНИЧЕНИЯ (ОСОЗНАННЫЕ ДЛЯ MVP)

- Нет автоплатежного шлюза (Stripe/PayPal/эквайринг) — платежи вручную через администратора.
- Нет полноценной бухгалтерской/банковской интеграции вывода.
- Нет enterprise-инфраструктуры (уровня CI/CD + production-hardening в полном объеме).

Это не ошибка реализации, а выбранная стратегия MVP:
быстро выйти в рабочий цикл, показать ценность и закрывать первые реальные сделки.

11) ПЛАН ПОСЛЕДУЮЩЕГО РАСШИРЕНИЯ (ПОСЛЕ MVP)

Этап 2 (после подтверждения спроса):
- подключение автоматического приема платежей,
- автоматические выплаты исполнителям,
- расширенный финансовый кабинет админа,
- усиление мониторинга, аудита и отказоустойчивости.

12) ФАКТИЧЕСКИЙ СТАТУС ПРОЕКТА СЕГОДНЯ

MVP функционально завершен для демонстрации и пилота:
- полный цикл "заказ -> отклик -> эскроу -> сдача -> выплата -> комиссия" работает.
- админ-модель ручных пополнений реализована.
- RU-интерфейс и демонстрационный сценарий готовы.

Проект готов к демонстрации заказчику и запуску первых реальных операционных кейсов в формате MVP.
